# R11 — Reverse-engineered (as-built) requirements

**Requirement.** The system shall support recording requirements that describe
already-implemented work (a reverse-engineered, "as-built" spec), and shall mark
such a requirement `Done` once its implementation is found in the code, so that an
apply to an existing codebase captures what the code already does without leaving
implemented behavior parked as open requirements.

## Rationale
Applying to an existing project (G2) frequently means the behavior exists in code
before any spec does. The sweep (R14) reads that code. To make `reqs.md` a faithful
durable memory (G4) it must be able to hold requirements derived from existing
behavior. These are hobby projects that generally carry no test suite — the user
verifies by *using* the software and files a `B#` bug when something is wrong.
Requiring separate verification before an as-built requirement could be marked done
would leave every captured requirement open indefinitely, making the `Done` flag
meaningless noise. So an as-built requirement whose implementation is present in the
code is marked done; the code is the evidence, and defects found later are tracked
as bugs. This gives `Done` one consistent meaning — implemented and not known to be
broken — across forward and as-built requirements. Traces to **G1**.

## Acceptance criteria
Verifiable conditions; all must hold before R11 is marked done in `reqs.md`.

- [x] Module 1 defines how to record a requirement for behavior found in code
      during the sweep (as-built), traced to a goal like any other requirement.
- [x] An as-built requirement whose implementation is present in the code is marked
      `Done` ☑; it is not left open for want of a test (`Done` = implemented and not
      known to be broken, verified by use rather than necessarily by a test).
- [x] A defect later found in behavior captured as-built is recorded as a `B#` bug
      rather than by reopening the requirement.
- [x] As-built requirements record their provenance (derived from code, with the
      source location) so they are not mistaken for authored spec, and the apply
      report states they were inferred from code.
- [x] Recording as-built requirements respects R6: only behavior explicitly
      present in code is described; intent that is not evident in the code becomes
      an open question, not a requirement.

## Design notes
- No new table column is needed. Provenance lives in the item's `reqs/<ID>.md`
  detail file (an **As-built** note naming the source file/symbol); trivial items
  may carry an inline `(as-built)` tag in the requirement text.
- `Done` carries one meaning for all requirements — implemented and not known to be
  broken (verified by use, not necessarily a passing test). Problems become bugs,
  not reopened requirements.
