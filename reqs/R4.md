# R4 — Structure an existing project's scattered information

**Requirement.** On the first apply to an existing project, the system shall sweep
the project for information that already exists — however scattered — and
reorganize it into this file's conventions, confirming the result with the user
before writing.

## Rationale
Applying to a messy existing project is the main event: it gathers the mess into
the conventions and the installed `CLAUDE.md` rules keep new work tidy. Under G5
the gather/align is re-run on every apply (not once), so the working files stay
converged on the current definition. This is migration of what's already there,
not invention. Traces to **G1**.

## Acceptance criteria
Verifiable conditions; all must hold before R4 is marked done in `reqs.md`.

- [x] The sweep covers README/docs, an existing `CLAUDE.md`, `TODO`/`FIXME`/`HACK`
      comments, the issue tracker, and scattered `*.md` files.
- [x] Findings are mapped into the module's format (goals, requirements, bugs, todos).
- [x] The proposed result is presented with the source of each item and confirmed
      before writing, unless the user has pre-authorized proceeding (see B1).
- [x] Implied-but-ambiguous items become open questions rather than table rows.
- [x] Relocating/deduping/reformatting explicit info is done; fabrication is not.

## Idempotency
The sweep is **not** a one-time ingestion. Under G5 (R14–R16) apply re-runs the
whole-project scan and re-aligns the working file on *every* apply, not only when
it is absent or empty. Convergence — not "scan once then stop" — is what keeps
re-applies safe:
- [x] If `reqs.md` is absent or empty, it is created from the sweep.
- [x] If populated but not matching the conventions, it is restructured in place,
      losslessly — content relocated/reformatted, never dropped or renumbered. (R15, R16)
- [x] If already matching the conventions, it is left untouched (convergent). (R7)
